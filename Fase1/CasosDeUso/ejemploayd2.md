# Core del negocio  

### Plataforma integral para la gestión y logística del transporte de mercancías 

LogiTrans Guatemala, S.A. se centra en una plataforma integral que optimiza la gestión y logística del transporte de mercancías. La propuesta busca modernizar la operación mediante la centralización de información, la automatización de procesos y la mejora de la experiencia tanto para clientes como para empleados.

 ![alt text](./Diagramas/CDU001v4.png)

  * **Nota:** Para simplificar la representación del diagrama, los actores Área de Contabilidad, Encargado de Patio, Agente Financiero, Agente Operativo, Agente Logístico, Piloto, Certificador FEL, Departamento de Cobros y Supervisor Operativo se integraron en el actor general “Usuario del sistema”, debido a que comparten interacciones equivalentes dentro del caso de uso analizado. | Nombre            | Registrar usuario                                                                                                                                                                                                                                                                                                                        |
| Código            | CCDU001.1                                                                                                                                                                                                                                                                                                                                |
| Actores           | Gerencia                                                                                                                                                                                                                                                                                                                                 |
| Descripción       | Permite crear un nuevo usuario dentro del sistema, registrando sus datos generales, fiscales o de contacto cuando aplique, categoría de riesgo, credenciales iniciales y rol de acceso.                                                                                                                                                  |

| Postcondiciones   | El usuario queda creado en estado activo o pendiente según política interna.<br>Se genera registro de auditoría.<br>El usuario queda asociado a un rol inicial.                                                                                                                                                                          |
| Flujo principal   | 1. Gerencia selecciona la opción de registrar usuario.<br>2. El sistema muestra formulario de alta.<br>3. Gerencia ingresa datos de identificación, contacto, acceso y rol.<br>4. | Flujo alterno     | 1. El correo, usuario o identificador ya existe y el sistema rechaza el alta.<br>2. Faltan datos obligatorios y el sistema solicita completarlos.<br>3. | Reglas de negocio | No se permite duplicidad de usuario, correo o identificador principal.<br>Todo usuario debe quedar asociado a un rol.<br>La categoría de riesgo debe quedar registrada cuando corresponda al tipo de usuario corporativo.<br>El alta solo puede ser realizada por Gerencia.                                                              |
| Reglas de calidad | Validación inmediata de datos antes de guardar.<br>Confirmación clara del resultado del alta.<br>Registro automático en auditoría.<br>Interfaz simple para reducir errores de digitación.                                                                                                                                                |

## CDU 001.2 – Autenticar usuario

| Campo             | Detalle                                                                                                                                                                                                                                                                                                                               |
| ----------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |

| Código            | CCDU001.2                                                                                                                                                                                                                                                                                                                             |
| Actores           | Usuario del sistema                                                                                                                                                                                                                                                                                                                   |
| Descripción       | Permite validar las credenciales de acceso del usuario para autorizar su ingreso a la plataforma y cargar sus permisos de acuerdo con el rol asignado.                                                                                                                                                                                |
| Precondiciones    | El usuario existe en el sistema.<br>El usuario se encuentra activo.<br>La plataforma está disponible.                                                                                                                                                                                                                                 |
| Postcondiciones   | Si la autenticación es exitosa, se abre sesión y se cargan permisos.<br>Si falla, se deniega el acceso y queda registro del intento.                                                                                                                                                                                                  |
| Flujo principal   | 1. El usuario ingresa usuario y contraseña.<br>2. El sistema valida las credenciales.<br>3. El sistema verifica que la cuenta esté activa.<br>4. | Reglas de calidad | Tiempo de respuesta objetivo menor a 8 segundos.<br>Mensajes de error claros pero sin exponer información sensible.<br>Auditoría obligatoria de accesos exitosos y fallidos.<br>Alta disponibilidad del servicio de autenticación.                                                                                                    |

# CDU 002 – Gestión de contrato digital
 ![alt text](./Diagramas/cdu002_uml_clasico.svg)

| Campo | Detalle |
| :---- | :------ |
| **Nombre** | Gestión de contrato digital |
| **Código** | CDU002 |
| **Actores** | Agente operativo, Cliente |
| **Descripción** | Permite al agente operativo crear un contrato digital para un cliente, integrando las condiciones comerciales, rutas autorizadas y el tarifario vigente, para que el cliente lo revise y firme electrónicamente, quedando el contrato confirmado y vigente en el sistema. |
| **Precondiciones** | Cliente previamente registrado en el sistema.<br>Agente operativo autenticado.<br>Cliente autenticado en el sistema.<br>Tarifario vigente disponible (CDU003 ejecutado previamente). |
| **Postcondiciones** | Contrato digital generado, firmado y confirmado con estado Vigente.<br>Contrato rechazado o con firma inválida según resultado del proceso. |
| **Flujo principal** | 1. El agente selecciona al cliente en el sistema.<br>2. El agente ingresa las condiciones comerciales del contrato (límite de crédito, plazo de pago y demás términos). *«include» Definir condiciones del contrato*<br>3. El agente selecciona las rutas autorizadas y los tipos de carga permitidos para el cliente. *«include» Definir rutas y tipos de carga*<br>4. El agente selecciona la opción Crear contrato.<br>5. El sistema integra las condiciones, rutas y el tarifario vigente. *«include» CDU003 – Gestión de tarifario*<br>6. El sistema genera el documento digital y lo guarda en estado Pendiente de firma.<br>7. El cliente accede al contrato, revisa condiciones y tarifas, y firma digitalmente.<br>8. El sistema recibe la firma y verifica su autenticidad e integridad.<br>9. El sistema confirma la validez de la firma y cambia el estado del contrato a Vigente.<br>10. Ambas partes reciben notificación de confirmación. |
| **Flujos alternos** | **FA1: Campos obligatorios vacíos al definir condiciones.**<br>FA1.1 El sistema muestra notificación de error indicando los campos faltantes.<br>FA1.2 El agente completa la información requerida.<br>FA1.3 Continúa en el paso 2 del flujo principal.<br><br>**FA2: Ruta seleccionada no existe en el sistema.**<br>FA2.1 El sistema notifica que la ruta no está registrada.<br>FA2.2 El agente corrige la selección.<br>FA2.3 Continúa en el paso 3 del flujo principal.<br><br>**FA3: Tarifario no disponible al generar el contrato.**<br>FA3.1 El sistema notifica la ausencia de tarifario vigente.<br>FA3.2 El agente gestiona la generación del tarifario mediante CDU003.<br>FA3.3 Continúa en el paso 4 del flujo principal.<br><br>**FA4: Cliente rechaza el contrato.**<br>FA4.1 El sistema marca el contrato como Rechazado.<br>FA4.2 El sistema notifica al agente operativo con el motivo indicado por el cliente.<br>FA4.3 El proceso finaliza.<br><br>**FA5: Firma digital inválida.**<br>FA5.1 El sistema detecta inconsistencia en la firma.<br>FA5.2 El sistema notifica al cliente y solicita una nueva firma.<br>FA5.3 Continúa en el paso 7 del flujo principal. |
| **Reglas de negocio** | El límite de crédito no puede ser negativo.<br>El plazo de pago debe corresponder a términos aprobados (15, 30 o 45 días).<br>Solo se pueden seleccionar rutas registradas en el sistema.<br>Debe seleccionarse al menos un tipo de carga.<br>Todo contrato debe tener un tarifario vigente asociado antes de generarse.<br>No puede generarse un contrato sin condiciones completas.<br>La firma digital implica aceptación total de los términos del contrato.<br>Solo firmas válidas permiten activar el contrato. |
| **Reglas de calidad** | Validación en tiempo real de campos obligatorios.<br>Generación del contrato en menos de 8 segundos.<br>Documento almacenado automáticamente tras su generación.<br>Visualización completa del documento antes de la firma.<br>Firma procesada y validada en menos de 10 segundos. |

# CDU 003 – Gestión de tarifario
 ![alt text](./Diagramas/cdu003_uml.svg)
| Campo | Detalle |
| :---- | :------ |
| **Nombre** | Gestión de tarifario |
| **Código** | CDU003 |
| **Actores** | Área de contabilidad |
| **Descripción** | Permite al área de contabilidad configurar y mantener el tarifario del sistema, definiendo las tarifas base según tipo de vehículo y carga, así como los montos por kilómetro, para que sean utilizados automáticamente al generar contratos digitales. De forma opcional, permite realizar ajustes o aplicar descuentos sobre las tarifas ya aplicadas a un contrato específico. |
| **Precondiciones** | Usuario del área de contabilidad autenticado en el sistema.<br>Tipos de vehículo y tipos de carga previamente registrados en el sistema. |
| **Postcondiciones** | Tarifario configurado y disponible para ser utilizado en la generación de contratos.<br>Tarifas ajustadas o descuentos aplicados registrados con historial de auditoría, en caso de haberse ejecutado la extensión. |
| **Flujo principal** | 1. El usuario de contabilidad accede al módulo de tarifario.<br>2. Selecciona el tipo de vehículo y el tipo de carga correspondiente. *«include» Definir tarifas según vehículo y carga*<br>3. Ingresa la tarifa base para esa combinación y guarda la configuración.<br>4. Selecciona el tipo de vehículo para configurar el costo por distancia. *«include» Configurar montos por kilómetro*<br>5. Ingresa el valor por kilómetro y guarda la configuración.<br>6. El sistema confirma el registro del tarifario y lo deja disponible como tarifario vigente. |
| **Flujos alternos** | **FA1: Campos vacíos o valores inválidos al ingresar tarifas.**<br>FA1.1 El sistema notifica el error indicando el campo con problema.<br>FA1.2 El usuario corrige el dato ingresado.<br>FA1.3 Continúa en el paso 3 o 5 del flujo principal según corresponda.<br><br>**FA2: Ajuste de tarifas o aplicación de descuentos** *(«extend»)*<br>FA2.1 El usuario accede a un contrato que ya tiene tarifario aplicado.<br>FA2.2 Selecciona la opción de ajuste e ingresa el porcentaje o monto de descuento.<br>FA2.3 El sistema recalcula los valores automáticamente y muestra el precio original junto al precio ajustado.<br>FA2.4 El usuario guarda la modificación.<br>FA2.5 El sistema registra el ajuste en el historial de auditoría.<br>FA2.6 Si el descuento excede el límite permitido, el sistema bloquea la operación y solicita al usuario que ajuste el porcentaje antes de continuar. |
| **Reglas de negocio** | Las tarifas deben ser valores positivos.<br>Debe existir al menos una tarifa configurada por tipo de vehículo.<br>El monto por kilómetro no puede ser negativo ni estar asociado a un vehículo inexistente.<br>Los descuentos no pueden superar el porcentaje máximo definido por la empresa.<br>Todo ajuste o descuento debe quedar registrado en el historial de auditoría.<br>Los cambios en el tarifario general no afectan contratos ya confirmados. |
| **Reglas de calidad** | Validación automática de campos en tiempo real.<br>Tiempo de guardado de configuración menor a 10 segundos.<br>Recálculo automático e inmediato al aplicar ajustes o descuentos.<br>Registro de auditoría obligatorio ante cualquier modificación. |

#### CDU004.1 - Generar Orden de Servicio

| Campo             | Detalle                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| ----------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Nombre            | Generar Orden de Servicio                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| Código            | CDU004.1                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| Actores           | Cliente Corporativo (Principal), Agente Operativo (Secundario)                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| Descripción       | Permite al cliente corporativo solicitar un movimiento de carga especificando origen, destino, tipo de mercancía y peso estimado. El sistema valida el contrato vigente (CDU002), verifica crédito disponible y facturas pendientes, y asigna la tarifa según el tarifario activo (CDU003). Si todo es válido, crea la orden con estado "Pendiente" y notifica al agente operativo.                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| Precondiciones    | El cliente debe estar autenticado (CDU001).<br>Debe existir un contrato vigente asociado al cliente (CDU002).<br>Debe existir un tarifario activo en el sistema (CDU003).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| Postcondiciones   | Orden de servicio creada con estado "Pendiente" y número único asignado, O Orden bloqueada con notificación al cliente.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| Flujo principal   | 1. El cliente accede al módulo de órdenes de servicio.<br>2. El cliente ingresa datos de la carga: origen, destino, tipo de mercancía y peso estimado.<br>3. El cliente solicita generar la orden.<br>4. El sistema consulta el contrato vigente del cliente en CDU002.<br>5. El sistema valida crédito disponible y facturas pendientes.<br>6. El sistema consulta CDU003 y asigna la tarifa automáticamente según tipo de carga y ruta.<br>7. El sistema genera un número único de orden.<br>8. El sistema crea la orden con estado "Pendiente".<br>9. El sistema notifica al agente operativo sobre la nueva orden.<br>10. La orden queda registrada y disponible para que CDU005 inicie la planificación.                                                                                                                                       |
| Flujos alternos   | **FA1: Contrato No Vigente**<br>FA1.1 El sistema bloquea la creación de la orden automáticamente.<br>FA1.2 El sistema notifica "Contrato no vigente, contacte a su agente".<br>FA1.3 La orden no se genera.<br><br>**FA2: Crédito Excedido**<br>FA2.1 El sistema bloquea la orden automáticamente.<br>FA2.2 El sistema notifica "Límite de crédito alcanzado".<br>FA2.3 La orden queda bloqueada hasta que se regularice el crédito.<br><br>**FA3: Facturas Vencidas**<br>FA3.1 El sistema bloquea la orden automáticamente.<br>FA3.2 El sistema notifica "Existen facturas vencidas. Regularice su cuenta".<br>FA3.3 La orden no puede generarse hasta saldar deuda.<br><br>**FA4: Tarifario No Disponible**<br>FA4.1 El sistema notifica "No existe tarifa configurada para esta ruta".<br>FA4.2 El agente operativo debe intervenir manualmente. |
| Reglas de negocio | - No se puede crear una orden sin contrato vigente (CDU002).<br>- No se puede crear una orden si el cliente excedió su límite de crédito o tiene facturas vencidas.<br>- La tarifa se asigna automáticamente desde CDU003; no puede ser modificada por el cliente.<br>- Cada orden debe tener un identificador único e irrepetible.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| Reglas de calidad | - La validación de contrato y crédito no debe exceder 5 segundos.<br>- La notificación al agente operativo debe ser inmediata al crear la orden.<br>- La respuesta al cliente (éxito o bloqueo) debe aparecer en menos de 8 segundos.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |

#### CDU004.2 - Modificar Orden de Servicio

| Campo             | Detalle                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| ----------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Nombre            | Modificar Orden de Servicio                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| Código            | CDU004.2                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| Actores           | Agente Operativo                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| Descripción       | Permite al agente operativo actualizar los datos de una orden que aún no ha tenido recursos asignados (estado "Pendiente" o "Aprobada"). Si el cambio afecta la tarifa o el contrato, el sistema re-consulta CDU002 y CDU003.                                                                                                                                                                                                                                                                                                                                                                                                     |
| Precondiciones    | La orden debe existir en el sistema.<br>La orden debe estar en estado "Pendiente" o "Aprobada".<br>Los recursos aún no han sido asignados (CDU005 no ha iniciado).                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| Postcondiciones   | Orden actualizada con los nuevos datos, O Modificación cancelada sin cambios.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| Flujo principal   | 1. El agente accede al módulo de gestión de órdenes.<br>2. El agente selecciona la orden a modificar.<br>3. El sistema muestra los datos actuales de la orden.<br>4. El agente actualiza los campos requeridos (origen, destino, tipo de carga, peso).<br>5. Si el cambio afecta la ruta o tipo de carga, el sistema re-consulta CDU003 y recalcula la tarifa.<br>6. El agente confirma los cambios.<br>7. El sistema actualiza la orden y registra el historial de modificación con usuario, fecha y campos modificados.<br>8. Si el cambio impacta a CDU005, el sistema notifica automáticamente a los departamentos afectados. |
| Flujos alternos   | **FA1: Orden con Recursos Ya Asignados**<br>FA1.1 El sistema bloquea la edición.<br>FA1.2 Notifica "La orden tiene recursos asignados y no puede modificarse".<br>FA1.3 Si es necesario modificar, el agente debe coordinar con CDU005 para liberar recursos primero.<br><br>**FA2: Nueva Tarifa No Disponible**<br>FA2.1 El sistema notifica "No existe tarifa para la nueva ruta".<br>FA2.2 El agente debe mantener la ruta original o escalar al área de contabilidad.                                                                                                                                                         |
| Reglas de negocio | - Solo se puede modificar una orden en estado "Pendiente" o "Aprobada".<br>- Toda modificación queda registrada en el historial (quién modificó, cuándo y qué campo).<br>- Si el cambio altera la tarifa, debe re-consultarse CDU003 automáticamente.                                                                                                                                                                                                                                                                                                                                                                             |
| Reglas de calidad | - El historial de cambios debe registrarse con timestamp y usuario responsable.<br>- La recalculación de tarifa no debe exceder 5 segundos.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |

El sistema verifica si aplica penalización según el contrato (CDU002).<br>5. El sistema solicita confirmación de cancelación al usuario.<br>6. El usuario confirma.<br>7. El sistema actualiza el estado de la orden a "Cancelada".<br>8. Si aplica penalización, el sistema consulta el contrato y registra el cargo correspondiente al cliente.<br>9. El sistema notifica a los departamentos afectados. |

| Reglas de calidad | - El sistema debe solicitar confirmación explícita antes de ejecutar la cancelación.<br>- El registro de la cancelación debe ser inmediato.                                                                                                                                                                                                                                                                                                                                                                                                                                |

### CDU 005 \- Planificar y asignar recursos

#### CDU 005.1 – Consultar recursos disponibles

| Campo | Descripción |
|-------|------------|
| **Nombre** | Consultar recursos disponibles |
| **Código** | CDU005.1 |
| **Actores** | Agente Operativo |
| **Descripción** | Permite al agente operativo visualizar las unidades de transporte y pilotos disponibles para la asignación de una orden de servicio, filtrando por disponibilidad, capacidad, ubicación y estado operativo. |
| **Precondiciones** | 1. La orden de servicio debe existir y encontrarse en estado “Aprobada” o “Pendiente de Asignación”. <br> 2. El agente operativo debe estar autenticado en el sistema. |
| **Post Condiciones** | 1. Se muestra una lista actualizada de vehículos disponibles. <br> 2. Se muestra una lista actualizada de pilotos disponibles. <br> 3. No se modifica el estado de la orden. |
| **Flujo principal** | 1. El agente operativo selecciona la orden de servicio. <br> 2. El sistema consulta las unidades de transporte en estado “Disponible”. <br> 3. El sistema consulta los pilotos sin órdenes activas. <br> 4. El sistema filtra los resultados según capacidad (tonelaje), sede y tipo de unidad. <br> 5. El sistema muestra la lista de recursos disponibles al agente. |
| **Flujos alternos** | FA1: No hay unidades disponibles <br> 1. El sistema muestra mensaje: “No existen unidades disponibles para esta sede o capacidad requerida”. <br> FA2: No hay pilotos disponibles  <br> 1. El sistema muestra mensaje: “No existen pilotos disponibles actualmente”. |
| **Reglas de negocio** | RN01: Solo pueden mostrarse vehículos en estado “Disponible”. <br> RN02: Un piloto no puede tener una orden en estado “Asignada” o “En Tránsito”. <br> RN03: La capacidad del vehículo debe estar dentro del rango parametrizado por contabilidad. <br> RN04: La consulta debe considerar la sede donde se originará la operación.|
| **Reglas de calidad** | RC01: El tiempo de respuesta de la consulta no debe exceder 5 segundos bajo carga normal. <br> RC02: La información mostrada debe estar sincronizada en tiempo real para evitar doble asignación. <br> RC03: La consulta debe registrarse en bitácora para fines de auditoría. |

#### CDU 005.2 – Asignar piloto y unidad a la orden
| Campo | Descripción |
|-------|------------|
| **Nombre** | Asignar Unidad y Piloto a la Orden |
| **Código** | CDU005.2 |
| **Actores** | Agente Operativo |
| **Descripción** |Permite al agente operativo asignar una unidad de transporte y un piloto disponible a una orden de servicio previamente creada, validando compatibilidad técnica, disponibilidad y cumplimiento de requisitos operativos. |
| **Precondiciones** | 1. La orden de servicio debe existir y estar en estado “Pendiente de Asignación”. <br> 2. Debe haberse consultado previamente la disponibilidad de recursos (CDU005.1). <br> 3. El agente operativo debe estar autenticado en el sistema. |
| **Post Condiciones** | 1. La unidad y el piloto quedan asociados a la orden. <br> 2. El estado de la orden cambia a “Asignada”. <br> 3. La unidad y el piloto cambian a estado “Asignados”. <br> 4. Se registra la asignación en bitácora. |
| **Flujo principal** | 1. El agente operativo selecciona una orden pendiente. <br> 2. El agente selecciona una unidad disponible. <br> 3. El agente selecciona un piloto disponible. <br> 4. El sistema valida compatibilidad técnica (peso, tipo de carga). <br> 5. El sistema valida documentación vigente del piloto. <br> 6. El sistema confirma la asignación. <br> 7. El sistema actualiza estados y registra la operación. |
| **Flujos alternos** | FA1: Unidad no compatible <br>  1. El sistema muestra mensaje: “La unidad seleccionada no cumple con la capacidad requerida”. <br>  2. El sistema solicita seleccionar otra unidad. <br>  <br>  FA2: Piloto con documentación vencida <br>  1. El sistema muestra mensaje: “El piloto no posee documentación vigente”. <br>  2. El sistema impide continuar hasta seleccionar otro piloto. <br> <br>  FA3: Recurso asignado simultáneamente <br>  1. El sistema detecta cambio de estado. <br>  2. Muestra mensaje de recurso no disponible. <br> |
| **Reglas de negocio** | RN01: La capacidad del vehículo debe ser igual o mayor al peso de la carga. <br>  RN02: El piloto debe poseer licencia vigente y permisos requeridos según el tipo de carga. <br> RN03: Un vehículo no puede tener más de una orden activa simultáneamente. <br>  RN04: Un piloto no puede estar asignado a más de una orden activa. <br>  RN05: La asignación solo puede realizarse en órdenes en estado “Pendiente de Asignación”.|
| **Reglas de calidad** | RC01: La validación de compatibilidad debe realizarse en tiempo real. <br>  RC02: El proceso de asignación debe completarse en menos de 5 segundos bajo condiciones normales. <br>  RC03: Todas las asignaciones deben quedar registradas en bitácora con fecha, hora y usuario responsable. <br>  RC04: El sistema debe prevenir condiciones de carrera para evitar doble asignación. |

# Drivers de restricción

### ERC 001 – Restricción presupuestaria y reutilización de infraestructura


| Campo                            | Detalle                                                                                                                                         |
| -------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------- |
| **Entorno**                      | La empresa busca modernizar su operación sin realizar una renovación completa de hardware desde la primera etapa.                               |
| **Restricción**                  | La arquitectura debe priorizar el uso de infraestructura existente y evitar soluciones con alto costo inicial o licenciamientos innecesarios.   |
| **Impacto en la arquitectura**   | Limita la selección de tecnologías, obliga a una propuesta austera y favorece componentes reutilizables, modulares y de crecimiento progresivo. |
| **Decisión condicionada**        | No se debe diseñar una solución que dependa desde el inicio de nueva infraestructura costosa para poder operar correctamente.                   |

# Justificación de decisiones arquitectónicas 
El sistema LogiTrans se desarrollará bajo el enfoque de Arquitectura Orientada a Servicios (SOA), ya que este estilo permite organizar el sistema en servicios de negocio independientes que facilitan la integración, la reutilización y la interoperabilidad entre los distintos procesos involucrados en la gestión logística de transporte de mercancías.

## Justificación:

### Modularidad del sistema
- La arquitectura SOA permite dividir el sistema en servicios independientes, cada uno responsable de una funcionalidad específica del dominio logístico.
- En LogiTrans se identifican servicios como: gestión de usuarios, gestión de contratos, gestión de tarifario, gestión de órdenes de servicio, planificación de recursos, gestión de carga de mercancía, traslado y entrega, facturación y pagos, y generación de reportes.
- Cada uno de estos servicios encapsula una parte de la lógica del negocio y puede evolucionar o mantenerse de forma independiente.

### Interoperabilidad
- Los servicios se exponen mediante interfaces estándar basadas en APIs REST, lo que permite que distintos componentes del sistema puedan comunicarse entre sí de manera uniforme.
- Además, este enfoque facilita la integración con sistemas externos, como el Certificador FEL.

### Reutilización de servicios
- La lógica de negocio implementada en cada servicio puede ser utilizada por diferentes módulos del sistema sin necesidad de duplicar funcionalidades.

### Escalabilidad y mantenimiento
- Aunque SOA no requiere el mismo nivel de granularidad que una arquitectura de microservicios, permite escalar servicios específicos según la demanda del sistema.
- Además, las modificaciones en un servicio pueden realizarse sin afectar directamente al resto del sistema.

### Alineación con la naturaleza del dominio logístico
- El proceso logístico involucra múltiples etapas coordinadas, como la gestión de contratos, la planificación de recursos, el manejo de carga, el traslado de mercancías y la facturación del servicio.
- La arquitectura SOA permite modelar cada uno de estos procesos como servicios autónomos pero interconectados, reflejando la estructura operativa de la empresa y facilitando la gestión de las operaciones.

### Preparación para evolución futura
- La adopción de SOA proporciona una base arquitectónica flexible que facilita la evolución del sistema en el futuro.
- En caso de requerirse mayor escalabilidad o independencia entre servicios, el sistema podría evolucionar hacia una arquitectura de microservicios.

## Decisión: Por qué escogimos SOA
- Balance entre modularidad y simplicidad: permite dividir el dominio en servicios sin el overhead de microservicios.
- Menor complejidad operativa: reduce la necesidad de pipelines CI/CD múltiples y herramientas de observabilidad avanzadas para el alcance académico.
- Reutilización y gobernanza: facilita compartir servicios comunes y mantener contratos uniformes entre equipos.
- Camino de evolución claro: si el sistema requiere mayor granularidad en producción, se podrá migrar gradualmente a microservicios.
