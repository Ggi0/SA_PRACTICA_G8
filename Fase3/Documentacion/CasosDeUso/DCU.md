## Casos de Uso Expandidos

## CDU001: Autenticación y Gestión de Usuarios

Sus expandidos serían:

- CDU001.1: Registro de usuario
- CDU001.2: Iniciar sesión
- CDU001.3: Cerrar sesión
- CDU001.4: Control de intentos y bloqueo temporal
- CDU001.5: Modificación de datos personales
- CDU001.6: Cambio de contraseña
- CDU001.7: Gestión de usuarios (Administrador)
- CDU001.8: Validación de sesión y rol

| Código            | CDU001.2                                                                                                                                                                                                                                                                   |
| Actores           | Usuario  Administrador                                                                                                                                                                                                                                    |
| Descripción       | Permite que un usuario registrado acceda a la plataforma mediante autenticación con sus credenciales, obteniendo acceso según su rol asignado.                                                                                                                             |
| Precondiciones    | - El usuario debe existir en el sistema (CDU001.1).                                                                                                                                                                                                                        |

El sistema valida las credenciales ingresadas.<br>5. El sistema genera un JWT con los claims esenciales del usuario: id, nombre, correo y rol.<br>6. El sistema retorna el token al frontend.<br>7. El frontend almacena temporalmente el token y lo envía en rutas protegidas mediante Authorization: Bearer <token>.<br>8. El sistema redirige al usuario según su rol (usuario general al catálogo, administrador al panel de gestión). |
| Flujos alternos   | **FA1: Credenciales inválidas**<br>FA1.1 El sistema notifica "Correo o contraseña incorrectos".<br>FA1.2 El sistema incrementa el contador de intentos fallidos.<br>FA1.3 El usuario puede reintentar.<br><br>**FA2: Usuario bloqueado temporalmente**<br>FA2.1 El sistema detecta que el usuario superó el límite de intentos.<br>FA2.2 El sistema notifica el bloqueo y el tiempo restante de espera.<br>FA2.3 El sistema deniega el acceso. |
| Reglas de negocio | - Si las credenciales no son válidas, el acceso es denegado.<br>- El sistema no debe revelar si el error es de correo o de contraseña para proteger la seguridad. |
| Flujo de excepción | **FE1: Error del sistema de autenticación**<br>FE1.1 El sistema de autenticación (base de datos o servicio externo) no responde.<br>FE1.2 El sistema notifica: "Error del sistema. Inténtelo de nuevo más tarde".<br>FE1.3 El sistema registra el error y alerta al equipo de infraestructura.<br><br>**FE2: Token de sesión no se genera**<br>FE2.1 Tras validar credenciales correctas, el sistema no puede generar el token de sesión.<br>FE2.2 El sistema notifica al usuario que ocurrió un error y solicita reintentar.<br>FE2.3 El contador de intentos fallidos no se incrementa. |
| Reglas de calidad | - La validación debe procesarse de forma segura.<br>- No se debe revelar información sensible en los mensajes de error.<br>- El inicio de sesión no debe exceder 2 segundos. |

| Actores           | Usuario, Administrador                                                                                                                                                                                                                                                                 |
| Descripción       | Controla los intentos fallidos de inicio de sesión y aplica un bloqueo temporal cuando se supera el límite permitido, protegiendo la plataforma de accesos no autorizados.                                                                                                            |

Si no supera el límite, el sistema permite un nuevo intento e informa los intentos restantes.<br>5. Si supera el límite, el sistema aplica un bloqueo temporal.<br>6. El sistema notifica al usuario el bloqueo y el tiempo de espera. |
| Flujos alternos   | **FA1: Límite de intentos alcanzado**<br>FA1.1 El sistema activa el bloqueo temporal.<br>FA1.2 El sistema deniega el acceso mostrando el tiempo restante de bloqueo.<br><br>**FA2: Error al registrar el intento**<br>FA2.1 El sistema deniega el acceso como medida de seguridad.<br>FA2.2 El sistema muestra un mensaje de error genérico. |
| Reglas de negocio | - El límite de intentos fallidos es de 5 intentos consecutivos.<br>- Al exceder el límite se bloquea el acceso por un tiempo definido (mínimo 15 minutos).<br>- El contador se reinicia tras un inicio de sesión exitoso. |

| Reglas de calidad | - El registro de intentos debe ser confiable y persistente.<br>- El bloqueo debe aplicarse de forma inmediata al cumplirse la condición.<br>- El mensaje de bloqueo debe indicar el tiempo de espera restante. |

| Precondiciones    | El usuario debe haber iniciado sesión correctamente. |
| Post Condiciones  | El acceso es permitido si el JWT es válido y el rol cumple con los permisos requeridos; de lo contrario, el acceso es denegado. |

| Precondiciones    | - El usuario tiene asientos bloqueados temporalmente (CDU004.2).<br>- El tiempo de reserva temporal no ha expirado.                                                                                                                                                                                                                                                                                                                                                             |

| Reglas de calidad | - El tiempo de espera visible para el usuario durante el procesamiento no debe exceder 5 segundos.<br>- Las transacciones fallidas deben poder reintentarse sin pérdida de datos.<br>- Los datos de pago deben transmitirse y almacenarse de forma segura y cifrada. |
